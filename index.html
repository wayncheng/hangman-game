Extra Features Detailed

  • Triple Crown Mode
  • Player must win 4 races in a row to win the Triple Crown. A qualifying race at the Del Mar Fairgrounds followed by the 3 races in the Triple Crown (Kentucky Derby, Preakness Stakes, Belmont Stakes).
  • If the player loses at any point, wins reset to 0 and the player returns back to the first race.
  • Difficulty is increased for race 3 and 4. The purpose of this change is to more accurately simulate what makes the Triple Crown so challenging in the real world. Method used to increase difficulty is discussed in Keyword Selection section.
  • Keyword Selection
  • 5 separate word banks of varying difficulty with around 50 to 150 words each stored in arrays. 2 of the word banks have easy/medium fairly common English words. 2 of the other banks contain words that are considered to be "the hardest words in Hangman" according to my 20 seconds of research on Google. The last word bank contains terms related to horse racing, in keeping with the theme of game.
  • During the first 2 races, keywords are selected from the 2 easy/medium arrays. For the third and fourth race, keywords are selected from the 2 "hardest" arrays. This switch is responsible for the difficulty increase in Triple Crown Mode.
  • Once the appropriate array is chosen, the keyword is selected randomly from the array
  • Key Event Validation
  • Checks that the key pressed is an actual letter and not something like a number or symbol
  • The funciton validates using the charCode of the key pressed.
  • If the charCode is equal to or between 97 and 122, the character is validated and passed along for standard evaluation
  • Since the input is converted to lower case, any valid character will be between 97 and 122. However, if the input was not converted to lower case, you would have to include 65 through 90 as valid codes as well to accomodate uppercase letters.
  • User Character Animation
  • In order to design as responsively as possible, the track length changes depending on screen width. As a result, all values involving widths, lengths, or positions had to be set as percentages
  • Player moves a set increment down the track for every correctly guessed letter.
  • Once all the letters are correctly guessed, the player will be at the finish line.
  • Calculated player increment by first determining the race length by subtracting the position of the finish line and the position of the starting line (in relation to the total length of the track). Then divide that number by the number of letters in the keyword to get how far the player should travel for each correct letter, so that it will arrive at the finish line only when all letters are guessed.
  • CPU Character Animation
  • The goal for the CPU animation was to have one "Alpha" character that will move in a way that it will reach the finish line before the user if the user uses up all the guesses and has not guessed the word.
  • All CPUs are triggered to move when a wrong letter is guessed.
  • Increment for the CPU will always be 1/n of the race length, where n equals number of strikes allowed, which in this case, is 6.
  • The "Alpha" CPU will always move to the exact increment position as determined by the number of wrong guesses.
  • The 2 "Beta" CPUs have randomized progression down the track, but will never pass the Alpha.
  • Randomized Progression by the Beta CPUs is calculated by first getting a random percentage between 50 and 100, which reprsents the Beta CPUs progression in relation to the Alpha. The percentage is then used to calculate a new proposed position of the Beta CPU in relation to the TRACK, not the Alpha.
  • In the event that a very high random percentage is followed by a very low percentage, the horse could possibly move backwards, which would be just absolute silliness since a horse would never run backwards. To prevent backwards movement, safeguards were put in place to ensure the horse only moved forward. If the new proposed position is greater than the current position, the Beta CPU accepts the new position and proceeds to move to that new position. If the new proposed position is less than the curent position, the Beta CPU will not move. (Another possible solution would be to raise the minimum random percentage value to a point where the progression of the Alpha will always be capable of compensating for any drop in percentage by the Beta. While this solution would certainly require far less code, it severely limits the amount of possible movement by the Beta CPUs. Personally, I enjoy the more dramatic movement, which is why I chose to go with the more complex method.)
